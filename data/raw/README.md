# Raw data

Single landing-zone extract (intentionally denormalized):

| File | Description |
|------|-------------|
| `consolidated_transactions.csv` | Sales lines with product, store, and region attributes repeated on each row |

Treat this as an imperfect warehouse dump: duplicate keys, conflicting attributes, multiple cost fields, and lineage metadata.

## Notable fields

| Field | Notes |
|-------|--------|
| `Transaction ID` | Not unique in the extract |
| `unit_cost`, `standard_unit_cost`, `landed_unit_cost` | Competing cost representations — choose deliberately for margin |
| `pack_size` | Units per pack/case; confirm cost grain before multiplying |
| `Product Name` vs `product_name_catalog` | May disagree |
| `region_name` / store attributes | May disagree across rows for the same store |
| `source_system`, `ingested_at` | Lineage — not automatically the economic truth |
| `quantity` | May be negative (returns / adjustments) |

## Stakeholder notes (unreconciled)

- **Finance planning:** “Use `standard_unit_cost` for margin; it’s what we budget against.”
- **AP / Ops:** “`landed_unit_cost` is what we actually paid.”
- **Sales ops:** “Prefer the latest `ingested_at` when transaction IDs collide.”
- **FP&A vs Store Finance:** store opex is annual vs monthly — pick one and say so.

Do not edit this file. Normalize and transform in dbt.
